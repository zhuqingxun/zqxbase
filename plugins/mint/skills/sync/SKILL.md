# MINT 下游同步（Sync）

检测用户对某个文件的手动修改，将变更传播到下游所有派生文件——人工改一处，AI 同步全链路。

## 用法

```
/mint:sync <被修改的文件路径或阶段>
```

示例：
- `/mint:sync D:\WORK\访谈记录\张三\03_校对稿\张三_校对稿.md` — 同步校对稿的修改到编辑稿和分析稿
- `/mint:sync 校对稿` — 同步校对稿的修改（自动定位工作目录）
- `/mint:sync 编辑稿` — 同步编辑稿的修改到分析稿

## 与 revise/patch 的区别

| | mint:patch | mint:revise | mint:sync |
|---|---|---|---|
| 触发方式 | 用户说"把XX改成YY" | 用户描述修订意图 | 用户已手动改好文件 |
| 变更来源 | 用户指令 | 用户指令 | 文件 diff |
| 怎么改 | 机械替换 | AI 语义理解 | AI 检测差异 + 适配传播 |

## 工作流程

### 第零步（可选）：预备份

如果用户**还没开始改**，可以先执行预备份：
```
/mint:sync --prepare 校对稿
```
这会将当前版本复制到 `old/` 子目录（如 `old/{name}_校对稿_v{N}.md`），然后用户去编辑器中修改，改完后再执行 `/mint:sync 校对稿`。

**版本号确定规则**：扫描 `old/` 目录中已有文件的最大版本号 +1。如果 `old/` 为空，版本号为 v1。

**强烈建议**在手动编辑前执行此步骤，可获得精确 diff。

### 第一步：定位被修改的文件

1. 如果用户提供了完整文件路径 → 直接使用
2. 如果用户只说了阶段名（"校对稿"/"编辑稿"） → 结合工作目录定位
3. 不确定时 → 通过 AskUserQuestion 让用户指定

确定文件后，推断所在阶段和工作目录。

### 第二步：检测变更内容

**方式 A（精确）：有 old/ 历史版本**

如果 `old/` 子目录中有该文件的上一个版本（通过 `--prepare` 或之前流水线自动生成）：
1. 使用 `diff --unified=3` 命令对比当前版本和 old/ 中最新的历史版本
2. 提取所有变更点

> **大文件策略**：校对稿通常超过 Read 工具的 token 限制（10000 tokens），**禁止**先尝试 Read 两个完整文件再 fallback。应直接用 `diff` 命令。若 diff 输出超大（>50KB），先 `diff --stat` 评估规模，再分段读取。
3. 将变更分类：
   - **删除**：用户删掉了哪些段落/句子
   - **修改**：用户改了哪些表述
   - **新增**：用户补充了哪些内容（少见但可能）

**方式 B（退化）：溯源检查——无历史版本时的兜底**

核心原则：**下游文件的每一段内容，都必须能在上游源文件中找到出处。找不到出处的内容 = 源头已被用户删除 = 下游也应删除。**

执行步骤：
1. 读取当前源文件（用户已手动修改过的版本）
2. 逐个读取下游文件
3. 对下游文件中的每个段落/观点/语录/摘要条目，检查：
   - **能溯源**：在源文件中能找到对应的内容片段 → 保留（如果表述有差异，适配更新）
   - **不能溯源**：源文件中找不到对应内容 → 标记为"源头已删除，建议删除"
   - **源文件有新增**：源文件中有内容但下游没有 → 标记为"源头新增，建议补充"
**溯源匹配策略**：按话题/段落主题匹配，而非逐字匹配。检查每个观点/论述的核心主题词和关键表述是否在源文件中出现。对于分析稿（摘要、行动项等），检查其引用或总结的内容是否仍有源文件依据。

4. 通过 AskUserQuestion 展示溯源检查结果，让用户确认：
   - 列出所有"建议删除"和"建议补充"的条目
   - 选项："确认，执行同步" / "需要调整，我补充说明"

**溯源检查特别适合删除场景**：用户删掉了校对稿中某段内容，下游编辑稿/分析稿中对应的内容自然找不到出处，会被正确识别并删除。

### 第三步：确定下游级联范围

与 revise 相同的级联逻辑：

| 被修改文件所在阶段 | 同步范围 |
|-------------------|---------|
| 03_校对稿 | 04_编辑稿 → 05_分析稿 |
| 04_编辑稿 | 05_分析稿 |
| 05_分析稿 | 无下游（提示用户无需同步） |

注意：同阶段的其他文件也可能需要同步。例如修改了 `_校对稿.md`，同阶段的 `_校对稿_脱敏.md` 也应同步更新。

**非 Markdown 派生文件的再生成**：扫描下游目录时，识别 `.docx`、`.mhtml`、`.pdf` 等非 Markdown 文件，纳入同步范围：
- 有对应 `.md` 源文件的（如 `_脱敏稿.docx` ↔ `_脱敏稿.md`）：在 `.md` 同步完成后，用 pandoc 等工具从 `.md` 重新生成
- 无对应 `.md` 源文件的（如独立的 `.mhtml` 导出）：在确认步骤中标注为"需手动更新"并说明原因
- 在确认步骤中明确列出所有需要重新生成的非 Markdown 文件

### 第四步：逐文件适配传播

对每个下游文件，根据文件类型适配变更：

| 源文件变更类型 | 编辑稿适配 | 结构化分析适配 | 精华原声适配 | 分析稿适配 |
|--------------|-----------|--------------|------------|-----------|
| **删除段落** | 删除对应的书面化内容 | 移除相关观点条目或原声引用 | 移除来源于被删段落的语录 | 更新摘要/发言人分析中的相关内容 |
| **修改表述** | 同步更新为新表述，保持编辑润色语感 | 更新观点描述和原声引用 | 更新相关语录 | 更新摘要/分析中的对应表述 |
| **新增内容** | 在对应位置补充书面化版本 | 判断是否需要新增观点条目 | 判断是否值得纳入精华原声 | 判断是否影响摘要/行动项/决策 |

**脱敏版本的处理**：如果同阶段存在脱敏文件，对脱敏文件执行同样的变更，额外确保不引入新的敏感信息。

**批量术语替换优化**：当变更中包含大量术语/人名修正时，按以下策略提升效率：
1. 对已确认的确定性替换，优先使用 `replace_all: true`
2. 按文件分组：同一文件的替换串行执行，不同文件的替换并行执行（最大化并行度）
3. 先处理全局替换（如 `A部门→B部门`），再处理需要精确匹配的局部替换

**预存问题扫描**：在逐文件替换过程中，如发现非本次修订引入但明显有误的内容（如 ASR 残留错字、前序阶段遗漏），收集到同步报告的「预存问题」段落中。不自动修复，由用户决定是否一并处理。

### 第五步：用户确认

传播完成前，通过 AskUserQuestion 展示变更预览：
- 列出每个下游文件将受到的影响
- 选项："确认同步" / "调整后再同步" / "取消"

### 第六步：执行同步并更新 meta.yaml

1. 对确认的文件执行 Edit
2. 在 `meta.yaml` 中追加修订记录：

```yaml
revisions:
  - timestamp: {ISO 时间戳}
    type: sync
    description: "从 {阶段名} 同步人工修改到下游"
    source_file: "{被修改的文件名}"
    files_affected: {受影响文件数}
```

### 第七步：输出同步报告

```markdown
## 同步报告

**源文件**：03_校对稿/{name}_校对稿.md
**检测到的变更**：删除 2 处、修改 3 处

| 下游文件 | 同步操作 |
|---------|---------|
| 03_校对稿/{name}_校对稿_脱敏.md | 同步删除 2 处 + 修改 3 处，脱敏规则已检查 |
| 04_编辑稿/{name}_编辑稿.md | 删除对应段落 2 处，更新表述 3 处 |
| 04_编辑稿/{name}_结构化分析.md | 移除 1 个观点条目，更新 2 条原声引用 |
| 05_分析稿/要点摘要.md | 更新 1 条摘要 |
| 05_分析稿/发言人分析.md | 无相关内容，跳过 |
```

**词表联动提示**：若变更中包含术语/人名修正（即 ASR 纠错类变更），在报告末尾提示：
> 本次同步包含 N 条术语/人名修正，建议执行 `/mint:patch` 将纠错沉淀到词表，避免未来转录重复出错。

## 边界规则

- **02_原始稿/ 绝对不改**：原始 ASR 输出是参照基准
- **old/ 不触碰**：历史归档保持原样
- **同步前必须确认**：与 patch/revise 不同，sync 涉及 AI 推断变更，必须让用户确认后再执行
- **只传播变更，不额外优化**：同步的目标是让下游文件反映源文件的变更，不顺手改其他内容
- **无法确定的变更要标注**：如果某个变更在下游文件中找不到对应内容，在报告中标注"无对应内容，跳过"
- **meta.yaml 修订记录必须更新**：每次 sync 操作都要在 revisions 中留痕

## 最后一步：更新元数据并输出引导块

1. **刷新 current.last_action + 计算 next_hints**：
   ```bash
   uv run --script {MINT_SCRIPTS}/meta_io.py refresh-last-action "<工作目录>"
   uv run --script {MINT_SCRIPTS}/meta_io.py compute-next-hints "<工作目录>"
   ```
   同时将 `current.cursor` 更新为 `"sync"`、`current.last_action_desc` 更新为 `"从{源阶段}同步到下游"`。

2. **"返回主流水线"语义覆盖**（PRD 7.3）：sync 已把修改传播到全部下游，手工覆盖 next_hints：

   - primary: `/mint:status <工作目录>` — 查看同步后的最终产出状态
   - alternatives: `/mint:patch`（若本次同步包含术语/人名修正，建议沉淀到词表）、`/mint:next`

3. **渲染引导块**：
   - Read `{MINT_REF}/next-hints-template.md`
   - 用覆盖后的 next_hints 填充 `{primary_cmd}` / `{primary_reason}` / `{alternatives_block}`
   - `{alternatives_block}` 按每行 `- {cmd}: {when}` 循环展开，空数组输出单行 `- 无`
   - 原样输出填充后的模板（保留开头的 `---` 分隔线）
