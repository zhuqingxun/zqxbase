# MINT 定向修订（Revise）

对锚定的清洁稿及其全部派生文件，执行内容层面的定制化修订。一处改动，全链路同步。

## 用法

```
/mint:revise <修订指令> [<工作目录>] [--from <起始阶段>]
```

参数说明：
- `<修订指令>`：修订内容描述
- `<工作目录>`：可选，MINT 工作目录路径。未指定时从对话上下文推断或询问用户
- `--from <起始阶段>`：可选，指定锚定文件所在阶段，修订从该阶段开始向下游级联。可选值：`校对稿`/`refine`（默认）、`编辑稿`/`polish`、`分析稿`/`extract`（同时支持中英文值）

示例：
- `/mint:revise 把"组织效能提升"改成"组织能力建设"` — 从校对稿开始，级联到编辑稿、分析稿
- `/mint:revise 删掉关于薪酬的敏感段落 --from 编辑稿` — 只改编辑稿和分析稿，不动校对稿
- `/mint:revise 把具体数字模糊化处理 --from 分析稿` — 只改分析稿

## 与 mint:patch 的区别

| | mint:patch | mint:revise |
|---|---|---|
| 改什么 | ASR 同音错字 | 内容措辞、段落取舍、视角调整 |
| 怎么改 | 机械替换（find-replace） | 语义理解后适配各文件 |
| 词表 | 更新 lessons-learned.md | 不涉及词表 |

## 工作流程

### 第一步：解析修订指令

从用户参数中理解修订意图，归类为：

- **措辞替换**："把 A 改成 B"——语义等价但换表达
- **内容删除**："删掉关于 XX 的部分"——移除特定话题/段落
- **脱敏调整**："把具体数字模糊化""隐去人名"
- **视角重构**："从管理者角度重新表述""语气更正式"
- **复合修订**：以上多种的组合

如果指令模糊（如"这段不太合适"），通过 AskUserQuestion 追问具体要怎么改。

### 第二步：定位工作目录与派生链

**定位工作目录**（优先级从高到低）：
1. 用户明确指定了目录 -> 用该目录
2. 当前对话上下文中有明确的工作目录 -> 用该目录
3. 用户提到人名或会议名 -> 尝试在常用工作路径下定位
4. 不确定时 -> 通过 AskUserQuestion 让用户确认

**锚定文件与下游级联**：根据 `--from` 参数确定锚定阶段，修订从该阶段开始向下游级联。

| --from 值 | 锚定文件 | 级联范围 |
|-----------|---------|---------|
| `校对稿`（默认） | `03_校对稿/{name}_校对稿.md` | 03 → 04 → 05 全链路 |
| `编辑稿` | `04_编辑稿/{name}_编辑稿.md` | 04 → 05（不动校对稿） |
| `分析稿` | `05_分析稿/` 下指定文件 | 仅 05（不动校对稿和编辑稿） |

**各阶段包含的文件**：

```
── 03_校对稿/ ──────────────────────────────
03_校对稿/{name}_校对稿.md           ← 锚定文件（--from 校对稿）
03_校对稿/{name}_校对稿_脱敏.md      ← 脱敏版本（如存在）

── 04_编辑稿/ ──────────────────────────────
04_编辑稿/{name}_编辑稿.md           ← 锚定文件（--from 编辑稿）
04_编辑稿/{name}_编辑稿_脱敏.md      ← 脱敏版本（如存在）
04_编辑稿/{name}_结构化分析.md
04_编辑稿/{name}_结构化分析_脱敏.md  ← 脱敏版本（如存在）
04_编辑稿/{name}_精华原声.md

── 05_分析稿/ ──────────────────────────────
05_分析稿/要点摘要.md
05_分析稿/发言人分析.md
05_分析稿/行动项.md
05_分析稿/决策与遗留.md
```

**智能推断起始阶段**：如果用户未指定 `--from` 但在指令中提到了具体文件（如"把编辑稿中的XX改掉"），自动推断锚定阶段。

**排除**：
- `02_原始稿/` 目录 — ASR 原始转录不改
- 任何 `old/` 子目录 — 历史归档不动

**注意**：`--from` 指定的是起始阶段，该阶段下的所有文件（不仅是锚定文件）都在修订范围内。例如 `--from 编辑稿` 时，编辑稿、结构化分析、精华原声都会被检查和修订。

**注意**：并非所有文件都一定存在（如未执行 Polish 则没有 04_编辑稿/ 文件）。只处理实际存在的文件。

### 第三步：逐文件适配修订

这是核心步骤——不同于机械替换，修订需要理解每个文件的上下文。

**对每个文件**：

1. **Read 全文**，理解该文件的结构和内容形式
2. **定位修订点**：找到与修订指令相关的段落/句子
3. **适配修改**：根据文件类型调整修订方式

#### 适配规则

| 文件类型 | 适配方式 |
|---------|---------|
| 清洁稿（clean） | 直接修改原文内容 |
| 编辑稿（polished） | 修改对应段落，保持编辑润色后的语感 |
| 结构化分析（structured） | 修改观点摘要中的对应表述；如引用了被改内容作为原声，同步更新 |
| 语录集（quotes） | 更新或移除相关语录条目 |
| 会议摘要（summary） | 修改对应要点的表述 |
| 发言人档案（speakers） | 更新发言人相关的观点或发言记录 |
| 行动项（actions） | 如修订影响到行动项内容，同步更新 |
| 决策记录（decisions） | 如修订影响到决策内容，同步更新 |
| 脱敏版本（`*_脱敏.md`） | 同步修订对应的脱敏文件；修改后检查是否引入了新的敏感信息（人名、职级、个人经历等），如有则按脱敏规则处理 |

#### 删除类修订的特殊处理

- **清洁稿**：删除对应段落，保持上下文连贯
- **编辑稿**：同步删除对应内容
- **结构化分析**：如果被删内容是某个观点的唯一支撑，整个观点条目可能需要移除或合并
- **语录集**：移除来自被删段落的语录
- **extract 文件**：如被删内容影响摘要/行动项/决策，同步更新或移除

### 第四步：更新 meta.yaml

在工作目录的 `meta.yaml` 中追加修订记录：

```yaml
revisions:
  - timestamp: {ISO 时间戳}
    type: revise
    description: "{修订指令概要}"
    files_affected: {实际修改的文件数}
```

### 第五步：输出修订报告

```markdown
## 修订报告

**修订指令**：{用户原始指令}

| 文件 | 修订操作 |
|------|---------|
| 03_校对稿/{name}_校对稿.md | 第 XX 行：将"XX"改为"YY" |
| 04_编辑稿/{name}_编辑稿.md | 第 XX 段：同步更新表述 |
| 04_编辑稿/{name}_结构化分析.md | 观点 X：同步更新原声引用 |
| 05_分析稿/要点摘要.md | 要点 X：更新摘要表述 |

### 未受影响的文件
以下文件不包含相关内容，无需修改：
- 05_分析稿/行动项.md
- 05_分析稿/决策与遗留.md
```

## 边界规则

- **02_原始稿/ 绝对不改**：原始 ASR 输出是参照基准
- **old/ 不触碰**：历史归档保持原样
- **修订范围精确**：只改与指令相关的内容，不顺手"优化"其他部分
- **结构化文件保持格式**：修改结构化分析时，保持原有的模板结构
- **语录集的完整性**：删除语录时确认引用来源标注同步清理
- **不确定就问**：如果修订指令可能导致内容歧义或大段删除，先通过 AskUserQuestion 确认再动手
- **meta.yaml 修订记录必须更新**：每次 revise 操作都要在 revisions 中留痕

## 最后一步：更新元数据并输出引导块

> **`{MINT_SCRIPTS}` / `{MINT_REF}` 路径约定**：分别指 mint 插件的 `scripts/` 和 `references/` 目录。首次引用时通过 `Glob` 定位对应目录下的任一文件，多结果时优先非 `marketplaces/` 路径。

1. **刷新 current.last_action + 计算 next_hints**：
   ```bash
   uv run --script {MINT_SCRIPTS}/meta_io.py refresh-last-action "<工作目录>"
   uv run --script {MINT_SCRIPTS}/meta_io.py compute-next-hints "<工作目录>"
   ```
   同时将 `current.cursor` 更新为 `"revise"`、`current.last_action_desc` 更新为 `"定向修订({--from 阶段})"`。

2. **"返回主流水线"语义覆盖**（PRD 7.3）：由于 revise 是非线性维护操作，compute-next-hints 的默认推荐可能偏移。根据本次修订的 `--from` 阶段和实际受影响文件，手工覆盖 next_hints 并写回 meta.yaml.next_hints：

   | --from | 推荐 primary | 备选 |
   |---|---|---|
   | 校对稿/refine | `/mint:polish <工作目录>` — 下游编辑稿需重新生成以反映修订 | `/mint:extract <工作目录>`、`/mint:status` |
   | 编辑稿/polish | `/mint:extract <工作目录>` — 下游分析稿需刷新 | `/mint:status` |
   | 分析稿/extract | `/mint:status` — 分析稿无下游，建议查看最终产出 | `/mint:next`、`/mint:patch` |

3. **渲染引导块**：
   - Read `{MINT_REF}/next-hints-template.md`
   - 用覆盖后的 next_hints 填充 `{primary_cmd}` / `{primary_reason}` / `{alternatives_block}`
   - `{alternatives_block}` 按每行 `- {cmd}: {when}` 循环展开，空数组输出单行 `- 无`
   - 原样输出填充后的模板（保留开头的 `---` 分隔线）
