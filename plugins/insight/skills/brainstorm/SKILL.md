# Brainstorm: 研究需求访谈

## 概述

通过结构化的访谈对话，与用户进行头脑风暴，分层、全面地澄清研究需求。对话上下文将作为 `/insight:biubiubiu` 的输入，也可选择将需求摘要保存为文件供后续流程使用。

## 核心原则

- **只谈需求，不谈技术方案**：聚焦于"研究什么"和"为什么研究"，不涉及"怎么实现"
- **访谈式对话**：以提问引导为主，帮助用户系统化思考，发现遗漏
- **分层深入**：从顶层愿景到具体细节，层层递进，确保完整性
- **自然流转**：层次之间不做刻意的阶段确认，保持对话的自然节奏

## 输入

用户提供的研究想法或方向：$ARGUMENTS

如果 $ARGUMENTS 为空，直接询问用户想要研究的方向。

## 访谈流程

### 第一层：愿景（Why & What）

围绕以下四个维度展开，每个维度通过 1-3 个关键问题引导：

**1. 要解决的问题**
- 当前对这个领域存在哪些认知盲区或疑问？
- 这个研究有多迫切？是否有时间节点？
- 现有的信息来源或已有认知是什么？为什么不够？

**2. 核心价值**
- 这个研究最独特的价值是什么？
- 完成研究后会获得什么核心收益？
- 一句话描述这个研究的价值主张

**3. 目标读者**
- 谁是主要读者？（角色、背景、专业水平）
- 是否有次要读者群体？
- 读者在什么情境下会阅读这个研究成果？

**4. 产出形态**
- 期望的产出载体是什么？（Markdown 报告、PPT、站点页面等）
- 是否有特定的深度偏好？（概览 / 深度分析 / 专家级）
- 期望的内容组织方式是怎样的？（按主题 / 按时间线 / 按对比维度等）

**引导方式**：不要一次性抛出所有问题。根据用户的初始描述，识别哪些维度已经隐含回答，只针对模糊或缺失的部分提问。每轮最多问 2-3 个问题，使用 AskUserQuestion 工具以选项形式呈现关键决策点。

### 第二层：场景（Who & Where）

在愿景明确后，自然过渡到内容模块梳理：

**1. 内容模块枚举**
- 梳理研究需要覆盖的主要模块（通常 3-7 个）
- 每个模块用一句话描述："{模块名}：研究 [什么内容]，回答 [什么问题]"
- 确认模块的优先级排序

**2. 深度与广度**
- 明确哪些模块需要深度挖掘
- 明确哪些模块只需概览（至少在第一版只需概览）
- 识别需要推迟到后续研究的方向

**3. 约束条件**
- 是否有时间限制？
- 是否有特定信息源偏好？（官方文档、学术论文、社区讨论等）
- 是否有已确定的排除范围？（不研究什么）

**引导方式**：基于愿景层的讨论结果，主动提出可能的内容模块让用户确认或补充。对于深度问题，可以用"这个模块是否需要深度分析？"的形式逐项确认。

### 第三层：细节（How it works）

对第二层梳理出的每个核心模块，逐一深入：

**针对每个模块讨论：**

**1. 内容模块细节**
- 这个模块包含哪些具体的子话题？
- 需要回答哪些关键问题？
- 是否有已知的重要概念或关键词？

**2. 信息源方向**
- 有哪些已知的权威信息源？
- 是否需要特定类型的素材？（架构图、数据表、案例分析等）
- 是否有推荐的 YouTube 频道、博客、或专家？

**3. 风险与盲区**
- 可能遇到的信息缺口有哪些？
- 哪些观点可能存在争议需要多方佐证？
- 是否有需要特别注意的偏见或误导信息？

**引导方式**：按模块优先级逐个讨论。每个模块讨论完毕后自然进入下一个。对于用户描述中明显遗漏的信息源方向，主动提出"是否需要关注 X 方面的信息？"

## 访谈技巧

### 提问策略

- **开放式引导**：先让用户自由描述，再针对性追问
- **具体化追问**：当回答模糊时，用"能举个具体的例子吗？"引导
- **反向验证**：用"所以你的意思是...对吗？"确认理解
- **补盲提醒**：主动提出用户可能忽略的维度（"你有没有考虑过...的方向？"）

### 对话节奏

- 每轮对话聚焦 1-2 个主题，避免信息过载
- 用户回答简短时，适当追问以获取更多细节
- 用户回答详细时，帮助提炼要点并确认
- 当某个话题讨论充分后，自然引入下一个话题

### 使用 AskUserQuestion 的场景

以下情况必须使用 AskUserQuestion 工具：
- 需要用户在多个研究方向中做选择时
- 需要确认研究边界（研究/不研究）时
- 需要确认模块优先级时

## 结束条件

当以下条件全部满足时，结束访谈：

1. 愿景层四个维度（问题、价值、读者、形态）均已明确
2. 内容模块已枚举并确认优先级
3. 研究边界（研究什么/不研究什么）已明确
4. 每个核心模块的内容细节、信息源方向、风险盲区均已讨论
5. 没有明显的遗漏或模糊点

## 结束输出

访谈完成后，输出结构化的需求摘要（仅展示在对话中，不保存文件）：

```
## 需求摘要

### 研究愿景
- **研究主题**：[一句话描述]
- **核心价值**：[一句话描述]
- **目标读者**：[读者画像]
- **产出形态**：[载体 + 深度级别]

### 内容模块（按优先级排序）
1. [模块名]：[一句话描述]
2. [模块名]：[一句话描述]
...

### 研究边界
- **深度挖掘**：[列表]
- **仅概览**：[列表]
- **明确不研究**：[列表]

### 约束条件
- [约束条件列表]

### 各模块研究要点
#### 模块1：[名称]
- 子话题：...
- 关键问题：...
- 信息源方向：...
（每个模块重复）
```

输出摘要后：
1. 询问用户是否有遗漏或需要补充的内容
2. 确认无遗漏后，使用 AskUserQuestion 询问下一步：
   - **启动全自主研究**：保存摘要文件后执行 `/insight:biubiubiu`，从摘要文件启动全自主研究团队
   - **保存摘要文件**：将需求摘要保存到 `research/brainstorm-summary-{topic-name}.md`（适合后续使用 `/insight:biubiubiu` 全自主研究，或需要跨会话保留需求记录的场景）
   - **两者都做**：先保存文件，再继续启动研究

如果用户选择保存文件，使用以下格式：

```markdown
---
description: "研究需求摘要: {topic-name}"
status: pending
created_at: {YYYY-MM-DDTHH:MM:SS}
updated_at: {YYYY-MM-DDTHH:MM:SS}
archived_at: null
---
```

文件内容使用上方「结束输出」中的需求摘要结构。保存路径：`research/brainstorm-summary-{topic-name}.md`。
