# PPT 精加工

将程序化生成的粗加工 PPT（python-pptx / pptxgenjs）升级为精加工版本。

## 为什么需要这个技能

粗加工 PPT 结构精准但视觉单调——单一字体、纯色填充、无图解辅助。精加工有两条路径，按需选择或组合使用：

- **纯代码增强**（python-pptx 后处理）：改善字体层级、渐变色、对齐精度、阴影质感。快速、确定性高、不依赖外部服务
- **NBLM 增强**（NotebookLM 视觉资产）：为关键页面生成信息图/概念图嵌入。视觉冲击力强，但每张图需审查

核心理念：架构师决定每一页该做什么、在哪里做，工具（python-pptx / NBLM）只是画笔。

## 前置条件

1. 粗加工 PPT 已存在（`generate_ppt.py` / `generate_ppt_v*.js` / 任何 python-pptx 或 pptxgenjs 生成的 PPTX）
2. 如选择 NBLM 增强路径：`notebooklm status` 已登录，且项目有可用 notebook（Phase 0 会自动处理）

## 执行流程

### Phase 0：环境准备（自动）

**纯代码增强**：确认 python-pptx 已安装（`uv add python-pptx`），跳到 Phase A。

**NBLM 增强**：需要 NotebookLM notebook。

1. 检查项目根目录的 `nblm-meta.json`
   - 存在且 notebook_id 有效 → `notebooklm use {id}` → 跳到 Phase A
   - 不存在 → 继续
2. `notebooklm status` 检查登录。未登录提示 `! notebooklm login`
3. 创建 notebook：`notebooklm create "{项目名} — PPT 精加工"`
4. 上传源材料（按优先级）：
   - `research/*.md`（调研素材，最重要）
   - `site/docs/{项目目录}/*.md`（网站内容）
   - 需求文档（`*.md`，如有）
   - 每个文件：`notebooklm source add --file "{路径}"`
   - 上限 50 个源，超出则跳过重复性高的
5. 保存 `nblm-meta.json`（含 notebook_id、sources_count、source_dirs）
6. `notebooklm status` 验证

### Phase A：审视与 Blueprint

1. **理解 PPT 结构**：读取生成脚本（`generate_ppt*.py` / `generate_ppt*.js`），提取：
   - 每页布局结构（图片页 / 表格页 / 纯文字页 / 图表页）
   - 色板（所有颜色常量）
   - 字体配置

2. **逐页判定增强策略**：

   | 标记 | 含义 | 动作 |
   |------|------|------|
   | `[KEEP]` | 已有官方图/截图，保持原样 | 不动 |
   | `[CODE]` | 用 python-pptx 增强（渐变、阴影、字体、对齐） | Phase C |
   | `[NBLM]` | 用 NotebookLM 生成视觉资产嵌入 | Phase B → C |

   **`[NBLM]` 判定三条件**（全部满足才标记）：
   - 页面有概念适合视觉化（流程、对比、关系）
   - 有可用空白或可收缩的现有元素
   - 视觉化显著提升信息传达效率（不仅是"更好看"）

   35 页 PPT 通常标记：25+ `[KEEP]`、5-8 `[CODE]`、3-5 `[NBLM]`

3. **Blueprint 文档**：

```yaml
meta:
  source_ppt: "output/xxx.pptx"
  output_ppt: "output/xxx-精加工.pptx"
  enhancement_mode: "hybrid"  # code-only / nblm-only / hybrid
  theme_colors:
    primary: "#1E2761"
    accent: "#E8553D"
    accent2: "#2B4C8C"
    bg: "#F4F6FA"

code_tasks:    # [CODE] 页面的增强清单
  - page: 1
    changes: ["标题加渐变填充", "法域标签加阴影"]

nblm_tasks:    # [NBLM] 页面的资产需求
  - page: 20
    position: "right-half"
    layout_change: "收窄左侧内容到 6 英寸"
    intent: "Meta 跨大西洋数据流可视化"
    prompt_seed: "flow diagram with arrows"

content_fixes:   # 内容清洗清单
  - page: 25
    issue: "包含内部站点 URL example.internal.com"
    fix: "替换为 '{项目名} | {年份}'"
```

   **`position` 取值**：`right-half` / `left-half` / `bottom-strip` / `bottom-card` / `center-full` / `top-banner`

4. **门禁**：向用户展示 blueprint 摘要，确认后继续。标错页面类型会导致后续全部浪费——这个确认成本低但防错价值高。

### Phase B：NBLM 串行生产

> 仅当 blueprint 包含 `nblm_tasks` 时执行。纯代码增强跳过此阶段。

对每个 NBLM 任务按顺序执行：

**B1. 撰写 3 个提示词变体。** 用英文，Part 1 以 "Draw a VERY SIMPLE" 开头（实测显著减少元素密度），Part 2 必须包含风格约束块。3 个变体应尝试不同视觉隐喻（韦恩图 vs 齿轮 vs 三角形），而非同一概念微调。

**风格约束块**（每个提示词必须附加）：
```
CRITICAL STYLE REQUIREMENTS:
- Pure white background
- NO title text above the diagram
- NO border or frame
- NO decorative elements or extra text descriptions
- Use only these colors: [从 theme_colors 选 2-3 个 hex]
- This image will be embedded in an existing PowerPoint slide - keep it minimal and clean.
```

每条约束的理由：白背景匹配 PPT 底色；无标题避免与 Python 标题重复；无边框让图片成为页面有机部分而非贴纸；限定颜色保持视觉一致。

**B2. 并行生成 3 候选。**
```bash
notebooklm generate infographic "{prompt}" --no-wait -n {id}
```
等待完成后下载。

**B3. 架构师审查。** Read 查看 3 张图，打分（概念准确 40% / 简洁度 30% / 风格兼容 20% / 可读性 10%）。≥ 7/10 通过。全不通过则改提示词再来 1 轮，最多 2 轮。兜底：Python 占位图 + TODO。

**B4. 沉淀提示词。** 保存到 `research/nblm-prompt-assets.md`，含通过和失败的，记录原因。

**提示词写作要诀**：好的提示词像素描指令（"画三个圆，标上这些字"），坏的像论文摘要。用英文写（NBLM 对英文否定指令理解更精准），图中标签会自动翻中文。翻译不准时显式双语标注（如 `labeled '企业软件 (Enterprise Software)'`）。

### Phase C：布局重构 + 组装

用 python-pptx 打开粗加工 PPT 进行修改。

**通用增强**（所有 `[CODE]` 和 `[NBLM]` 页面）：

```python
from pptx import Presentation
from pptx.util import Inches, Pt, Emu
from pptx.dml.color import RGBColor
from pptx.enum.text import PP_ALIGN

prs = Presentation('input.pptx')
slide = prs.slides[page_index]

# 遍历 shapes 修改属性
for shape in slide.shapes:
    if shape.has_text_frame:
        for para in shape.text_frame.paragraphs:
            for run in para.runs:
                run.font.size = Pt(11)
                run.font.color.rgb = RGBColor(0x2C, 0x33, 0x50)
```

**NBLM 图片插入**（`[NBLM]` 页面）：
1. 遍历 shapes，根据 blueprint `layout_change` 缩小/移动文字框腾出空间
2. 添加浅灰圆角矩形作为视觉容器
3. 插入图片到预留区域，不超出边界
4. 添加 "Generated by NotebookLM" 标注（7pt 斜体）

**pptxgenjs 兼容性注意**：pptxgenjs v4 生成的 PPTX 常有 OOXML 合规问题（如 Content_Types.xml 声明不存在的 slideMaster），可能触发 PowerPoint 修复提示。如果发现此问题，用 python-pptx 重新保存可修复大部分情况。另外，pptxgenjs 的 `barGrouping:"stacked"` 多系列图表会触发 PowerPoint 修复——避免使用。

### Phase D：独立审查（Agent Team）

精加工完成后，启动独立审查 agent 进行零信任验证。审查 agent 不参与生成过程，从零开始审查产出物。

**D1. 启动审查 agent**

使用 Agent 工具启动一个独立的审查 agent（`reviewer`），提供以下信息：
- 精加工 PPT 文件路径
- 原始粗加工 PPT 路径（用于对比）
- Blueprint 文档（哪些页面做了什么修改）
- LibreOffice soffice 路径（用于渲染）

**D2. 审查 agent 执行三轮检查**

**第一轮：程序化遮挡扫描**
- 用 python-pptx 遍历所有 slide 的所有 shape
- 检测文字 shape 与其他 shape 的包围盒重叠（>5% 面积）
- 排除设计性重叠（编号+背景、标题+副标题等相邻文本对）
- 输出遮挡报告：page / 文字内容 / 遮挡源 / 重叠百分比

**第二轮：内容清洗扫描**
- 扫描所有 slide 的所有文本，检查：
  - URL（http/https/域名模式）
  - 文件路径（C:/、D:/、/home/ 等）
  - 内部标记（TODO、FIXME、DEBUG 等）
  - 非目标语言残留（如中文 PPT 中的英文调试文本）
- 输出清洗报告

**第三轮：视觉渲染审查**
- 使用 LibreOffice 将精加工 PPT 转为 PDF
- 使用 PyMuPDF 将每一页（不是抽样，是每一页）渲染为 150 DPI JPG
- 对每张 JPG 用 Read 工具视觉检查：
  - 文字是否被遮挡
  - 图片是否在预留区域内
  - 元素是否对齐
  - 配色是否一致
- 输出视觉审查报告

**D3. 审查报告与修复循环**

审查 agent 将三轮报告合并为统一的审查报告，通过 SendMessage 发送给 Leader：
- PASS：所有检查通过，可进入 Phase E 交付
- FAIL + 问题清单：Leader 修复后重新提交审查（最多 2 轮）

**D4. 审查 agent 签收**

所有问题修复后，审查 agent 在报告中标记"APPROVED"，关闭。

### Phase E：交付

1. 用 PowerShell COM 自动化验证 PPTX 可正常打开（无修复提示）
2. 输出精加工报告：修改页面清单、使用策略、评分、提示词沉淀
3. 保留粗加工和精加工两个文件

## 可用的视觉资产类型

| 类型 | 命令 | 适合 | 注意 |
|------|------|------|------|
| 信息图 | `generate infographic` | 概念图、流程、对比 | **首选**，快且稳定 |
| 单页幻灯片 | `generate slide-deck` (1 页) | 风格化数据展示 | 需截取局部 |
| 思维导图 | `generate mind-map` → Python 渲染 | 关系网络 | JSON 格式需二次渲染 |
| 视频帧 | `generate video` → 截帧 | 场景化插图 | 最慢（5-10 分钟），慎用 |

## 实战经验（从失败中沉淀）

### NBLM 嵌入区域必须匹配图片宽高比

NBLM 信息图有自然宽高比（通常 16:9 或 4:3）。嵌入区域的宽高比必须与之接近，否则图片被压缩变形或文字不可读。**绝对禁止**将信息图塞进极端比例的窄条（如 12" x 1.5" = 8:1）。

实操规则：
- 嵌入宽度 ≤ 6" 时，高度 ≥ 宽度 x 0.5（最宽 2:1）
- 嵌入宽度 > 6" 时，高度 ≥ 3"（保证文字可读）
- 如果可用空间不满足上述条件，**不要嵌入 NBLM**，保留原始程序化图形

### 装饰元素必须避开内容区域

半透明装饰圆/线条即使 opacity 很低（4-6%），在 PowerPoint 实际渲染中仍会**遮挡文字**（尤其深色背景上的浅色文字）。

实操规则：
- 添加装饰元素前，**先扫描该区域是否有 text shapes**（包括表格、标签等）
- 装饰元素的包围盒不得与任何文字 shape 的包围盒重叠
- 封面/封底的装饰元素放在**内容区域下方**（y > 所有文字的 bottom），不要放在文字同一水平线

### 精加工 = 视觉增强 + 内容清洗

粗加工 PPT 的问题不仅是视觉单调，还常有内容瑕疵（非目标语言文本、内部 URL 泄露、卡片布局空间浪费）。Phase A Blueprint 阶段必须同时审视内容问题，在 Blueprint 中增加 `content_fixes` 清单，与 `code_tasks` / `nblm_tasks` 并列。

### 收窄内容时必须保护标题区域

NBLM 嵌入需要收窄左侧内容为右侧图片腾出空间。但 `shrink_content_for_nblm` 不能无差别收窄所有文本框——标题（y ≤ 1.2"）在图片上方，不会重叠，收窄反而导致长标题换行后与装饰条重叠。

实操规则：
- 只收窄 y > 1.2" 的文本框（bullet 内容区域）
- 标题、装饰条（y ≤ 1.2"）保持原宽度不动
- 收窄后验证：标题是否仍单行显示（用字符数 × 单字宽度估算）

### 精加工必须清洗内部 URL 和敏感信息

粗加工 PPT 常包含内部站点 URL、文件路径、调试标记等。这些在对外分享的精加工版本中必须移除。

实操规则：
- Phase A Blueprint 阶段扫描所有 slide 的文本，标记含有 URL（http/https/域名）、文件路径（C:/、D:/）、内部标记的内容
- 在 Blueprint 的 `content_fixes` 清单中列出需清洗的项
- Phase C 执行时替换或删除这些内容

## 异常处理

| 情况 | 处理 |
|------|------|
| NBLM 生成超时 | `artifact poll {task_id}` 轮询，最长 10 分钟 |
| 3 候选都不合格 | 改提示词重做 1 轮；仍不合格降级 Python 占位图 + TODO |
| 输出文件被锁 | 换文件名（加 v2/v3 后缀） |
| Notebook 无源 | Phase 0 自动处理 |
| 登录过期 | `notebooklm login` |
| PowerPoint 修复提示 | python-pptx 重新保存；检查是否使用了 stacked chart |
